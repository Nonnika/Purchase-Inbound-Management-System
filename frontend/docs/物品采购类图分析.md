## 五、系统业务模块划分

### 1. 用户与权限模块

涉及类：`roles`、`users`、`departments`

该模块实现了基于角色的访问控制（RBAC）体系。角色表定义系统权限集合，用户通过 `role_id` 关联角色获取对应权限；部门表采用自关联设计支持多级组织架构，用户通过 `department_id` 归属到具体部门。这种设计能够灵活地支持企业中常见的多部门、多角色的组织管理需求。

### 2. 物品与分类模块

涉及类：`item_categories`、`items`、`warehouses`

该模块是采购系统的核心数据管理部分。物品分类采用树形结构，支持无限层级的分类体系（如"办公用品 → 笔 → 圆珠笔"）；物品表记录了价格、库存、预警线等关键信息；仓库表管理物品的物理存放位置。三者通过外键关联形成完整的物品管理体系。

### 3. 库存冻结模块

涉及类：`items`、`frozen_items_users`、`users`

该模块实现库存冻结机制，是采购流程中防止超卖的关键设计。当用户发起采购申请时，系统在 `items` 表中扣减 `frozen_inventory`（而非 `item_inventory`），同时在 `frozen_items_users` 中创建一条关联记录。待采购确认出库后，再扣减实际库存并释放冻结记录。

### 4. 区块链溯源模块

涉及类：`items_block`、`items_chain`、`items`、`users`、`warehouses`

该模块引入区块链思想，为物品的每一次流转（入库、冻结、出库、调拨等）生成不可篡改的链式记录。`items_block` 表以 `parent_hash` 构成链式结构，`items_chain` 表记录每条链上操作的完整上下文（操作用户、仓库、库存快照等），实现了物品全生命周期的可追溯性。

## 六、设计亮点分析

1. **库存双轨管理**：通过 `item_inventory`（可用库存）和 `frozen_inventory`（冻结库存）的分离设计，实现了高并发场景下的库存防超卖机制，避免了传统"先扣后补"模式下的数据不一致风险。

2. **区块链溯源**：将区块链的哈希链式结构引入物品管理，使得每一次库存变动都具有不可篡改的审计记录，适用于对合规性要求较高的采购场景。

3. **树形结构复用**：`departments` 和 `item_categories` 均采用自关联的树形设计，以最小的表结构代价实现了灵活的层级管理，避免了硬编码层级或创建过多冗余表。

4. **关系解耦**：通过 `frozen_items_users` 中间表将"冻结"这一操作独立建模，使得冻结关系可追溯、可统计，同时也便于后续扩展（如批量冻结、部分释放等操作）。
